---
title: 플랫폼 통합
description: 하나의 Rust 코어가 얇은 바인딩과 @wvb/bridge 웹 레이어를 통해 Electron, Tauri, Android, iOS, Deno에 도달하는 방법.
---

Webview Bundle은 다섯 개가 아니라 하나의 시스템입니다. 포맷, 번들 소스, 프로토콜 핸들러, 원격 클라이언트, 업데이터, 무결성 및 서명 검사가 모두 하나의 Rust 크레이트 안에 있습니다. 모든 플랫폼은 얇은 바인딩을 통해 바로 그 크레이트를 사용합니다. 그래서 Electron용으로 빌드한 `.wvb`는 Tauri, Android, iOS, Deno에서도 똑같이 동작합니다. 개별 플랫폼 가이드를 읽기 전에 이 문서를 먼저 읽으세요. 각 가이드는 어떤 패키지가 어디에 속하는지 이미 안다고 가정합니다.

## 하나의 코어, 여러 호스트

Rust 크레이트 `wvb`([crates.io](https://crates.io/crates/wvb), 버전 0.2.0)가 단일 진실 공급원입니다. 이 크레이트는 다음을 구현합니다.

- `.wvb` 번들 포맷(헤더, 인덱스, LZ4로 압축된 데이터),
- 번들 소스(builtin 및 remote 디렉터리),
- 번들 프로토콜 핸들러와 로컬 프로토콜 핸들러,
- 원격 HTTP 클라이언트,
- 업데이터(확인, 다운로드, 검증, 설치),
- 무결성(SHA-2) 및 서명 검증.

각 바인딩은 의도적으로 얇습니다. 코어를 호스트 언어에 노출하고, 해당 호스트에 URL 스킴을 등록하며, 요청을 코어로 전달합니다. 코어를 수정하거나 기능을 추가하면 다음 바인딩 릴리스에서 모든 플랫폼에 반영됩니다. 전체 API는 [docs.rs/wvb](https://docs.rs/wvb)에서 확인하세요.

<Callout type="info">
  Rust 코어는 URL 스킴을 등록하거나 검증하지 않습니다. `app://`이나 `bundle://` 같은 스킴을 선택하고
  OS에 등록하는 일은 바인딩의 몫입니다. 요청이 파일로 매핑되는 방식은 [프로토콜
  처리](/docs/guide/protocol-handling)를 참고하세요.
</Callout>

## 전달 방식

각 플랫폼은 서로 다른 배포 채널을 통해 코어에 도달합니다.

| 플랫폼                   | 바인딩                    | 배포 방식                                                             |
| ------------------------ | ------------------------- | --------------------------------------------------------------------- |
| Electron / Node.js       | `@wvb/node`               | N-API 네이티브 애드온(NAPI-RS), npm에 플랫폼별 프리빌드 바이너리 제공 |
| Tauri(데스크톱 + 모바일) | `wvb-tauri` Rust 크레이트 | crates.io의 Tauri v2 플러그인, 버전 0.1.0                             |
| Android                  | `packages/ffi`(UniFFI)    | Kotlin 바인딩, `webview-bundle-android` 저장소에서 사용               |
| iOS                      | `packages/ffi`(UniFFI)    | Swift 바인딩, `webview-bundle-ios` 저장소에서 사용                    |
| Deno Desktop             | `@wvb/deno`               | 프리빌드 dylib 위의 Deno FFI(실험적)                                  |

### Electron과 Node.js

`@wvb/node`(0.1.0)는 코어를 프리빌드 바이너리가 포함된 N-API 네이티브 애드온으로 감쌉니다. 여느 npm 의존성처럼 설치하세요.

```sh
npm install @wvb/node
```

```ts title="main.ts"
import { readBundle, BundleSource } from '@wvb/node';

const source = new BundleSource({
  builtinDir: './bundles',
  remoteDir: './remote',
});
const bundle = await readBundle('./app.wvb');
```

Electron에서는 `@wvb/electron`(0.1.0)을 사용하세요. 이 패키지는 `@wvb/node` 위에 만들어졌으며, 스킴 권한, 프로토콜 등록, IPC 채널을 대신 연결해 줍니다.

```sh
npm install @wvb/electron
```

```js title="main.cjs"
const { app, BrowserWindow } = require('electron');
const { bundleProtocol, wvb } = require('@wvb/electron');

const instance = wvb({
  source: { builtinDir: path.join(__dirname, 'bundles') },
  protocols: [bundleProtocol('app', { onError: e => console.error('[wvb]', e) })],
});

app.whenReady().then(async () => {
  await instance.whenProtocolRegistered();
  const window = new BrowserWindow({
    webPreferences: { contextIsolation: true, nodeIntegration: false },
  });
  await window.loadURL('app://hacker-news.wvb');
});
```

electron-builder와 Electron Forge용 패키징 헬퍼인 `@wvb/electron-builder`와 `@wvb/electron-forge`는 번들 패킹을 빌드에 연결합니다. [Electron 가이드](/docs/guide/platforms/electron)와 [Node API 레퍼런스](/docs/references/node)를 참고하세요.

### Tauri

Tauri 통합은 Rust 크레이트 `wvb-tauri`(crates.io의 0.1.0)입니다. `@wvb/tauri` npm 패키지는 없습니다. 크레이트를 추가하고 Tauri v2 플러그인으로 등록하세요.

```toml title="src-tauri/Cargo.toml"
[dependencies]
wvb-tauri = "0.1.0"
```

```rust title="src-tauri/src/lib.rs"
use wvb_tauri::{Config, Protocol, Source};

tauri::Builder::default()
  .plugin(wvb_tauri::init(
    Config::new()
      .source(Source::new().builtin_dir("bundles"))
      .protocol(Protocol::bundle("bundle")),
  ))
  .run(tauri::generate_context!())
  .unwrap();
```

플러그인 명령어를 호출하는 프런트엔드는 capability 파일에 설정된 ACL 권한을 부여해야 합니다.

```json title="src-tauri/capabilities/default.json"
{
  "permissions": ["core:default", "wvb-tauri:default"]
}
```

같은 크레이트가 데스크톱과 모바일을 모두 처리합니다. Android에서는 builtin 번들이 APK 안에 `asset://` 리소스로 포함됩니다. 그래서 builtin 번들을 함께 배포하는 앱은 `tauri_plugin_fs`도 등록해야 합니다.

```rust
tauri::Builder::default()
  .plugin(tauri_plugin_fs::init()) // Android에서 builtin 번들에 필요함
  .plugin(wvb_tauri::init(config))
```

[Tauri 가이드](/docs/guide/platforms/tauri)를 참고하세요.

### Android와 iOS

Android와 iOS는 하나의 바인딩 생성기 `packages/ffi`를 공유합니다. 이는 [UniFFI](https://mozilla.github.io/uniffi-rs/)를 사용해 코어로부터 Kotlin과 Swift 바인딩을 생성하는 Rust 크레이트입니다. 빌드하면 세 개의 GitHub 릴리스 자산이 만들어집니다.

- `android.zip` — Android용 Kotlin 바인딩과 `jniLibs`,
- `apple.zip` — Apple 플랫폼용 Swift 바인딩과 정적 라이브러리,
- `WebViewBundleFFI.xcframework.zip` — iOS용 xcframework.

두 개의 전용 저장소가 이 자산들을 사용하며, 릴리스 태그 `ffi/<version>`으로 해석합니다.

```kotlin title="Android — dev.wvb 바인딩"
import dev.wvb.BundleSource
import dev.wvb.BundleSourceConfig

val source = BundleSource(
  BundleSourceConfig(
    builtinDir = "$filesDir/bundles",
    remoteDir = "$cacheDir/remote",
    builtinManifestFilepath = null,
    remoteManifestFilepath = null,
  ),
)
val bundle = source.fetchBundle("app")
```

```swift title="iOS — WebViewBundleLibrary 바인딩"
import WebViewBundleLibrary

let source = BundleSource(
  config: BundleSourceConfig(
    builtinDir: "\(docs)/bundles",
    remoteDir: "\(caches)/remote",
    builtinManifestFilepath: nil,
    remoteManifestFilepath: nil
  )
)
let bundle = try await source.fetchBundle(bundleName: "app")
```

```swift title="Package.swift"
.binaryTarget(
  name: "WebViewBundleFFI",
  url: "https://github.com/webview-bundle/webview-bundle/releases/download/ffi/0.1.0/WebViewBundleFFI.xcframework.zip",
  checksum: "<sha256>"
)
```

### Deno

`@wvb/deno`는 프리빌드 동적 라이브러리 위의 Deno FFI를 통해 코어에 도달합니다. GitHub Releases에서 dylib를 설치한 다음 로드하세요.

```sh
deno run -A jsr:@wvb/deno/install --out vendor/wvb
```

이 통합은 실험적입니다. [Deno 가이드](/docs/guide/platforms/deno)와 [Deno API 레퍼런스](/docs/references/deno)를 참고하세요.

## 브리지

위의 바인딩들은 네이티브 쪽에서 실행됩니다. 브리지 `@wvb/bridge`(0.1.0)는 웹 쪽입니다. 웹뷰 안의 JavaScript가 호스트와 상관없이 하나의 통일된 API로 네이티브 소스, 원격, 업데이터 작업을 호출하게 해 줍니다.

```sh
npm install @wvb/bridge
```

단일 진입점은 `invoke()`입니다.

```ts
import { invoke } from '@wvb/bridge';

const update = await invoke('updaterGetUpdate', { bundleName: 'app' });
```

`invoke()`를 이름으로 직접 호출하는 경우는 드뭅니다. 브리지는 네이티브 명령어를 세 개의 타입이 지정된 객체 `source.*`, `remote.*`, `updater.*`로 묶으며, 각 메서드는 타입이 지정된 `invoke()` 호출입니다.

```ts
import { source, remote, updater } from '@wvb/bridge';

const bundles = await source.listBundles();
const info = await updater.getUpdate('app');
if (info.isAvailable) {
  await updater.download('app');
  await updater.install('app', info.version);
}
```

- `source.*` — 번들 목록 조회, 파일 경로 해석, 메타데이터 읽기, 다운로드한 버전 관리.
- `remote.*` — 원격 서버에서 번들 목록 조회, 검사, 다운로드.
- `updater.*` — 업데이트 확인, 다운로드, 설치.

브리지는 런타임에 호스트를 감지하고 각 호출을 알맞은 전송 방식으로 라우팅합니다.

| 플랫폼   | 전송 방식                                               |
| -------- | ------------------------------------------------------- |
| Electron | `window.wvbElectron.invoke(name, params)`               |
| Tauri    | Tauri `invoke('plugin:wvb-tauri\|<command>', params)`   |
| Android  | `window.wvbAndroid.postMessage(...)`                    |
| iOS      | `window.webkit.messageHandlers.wvbIos.postMessage(...)` |

어디서나 동일한 `source`, `remote`, `updater` 메서드를 사용합니다. `@wvb/bridge` 클라이언트는 electron, tauri, android, ios를 지원합니다. Deno Desktop은 `@wvb/deno-desktop`을 통해 자체 바인딩을 제공합니다.

`@wvb/bridge/testing`을 임포트하면 네이티브 호스트 없이 웹뷰 코드를 테스트할 수 있습니다.

```ts title="app.test.ts"
import { mockBridge } from '@wvb/bridge/testing';
import { updater } from '@wvb/bridge';

using bridge = mockBridge({ platform: 'electron' });
bridge.mockInvoke('updater.getUpdate', () => ({
  name: 'app',
  version: '1.2.0',
  isAvailable: true,
}));

const info = await updater.getUpdate('app');
```

<Callout type="info">
  `@wvb/bridge/testing`은 `mockInvoke`, `mockPlatform`, `mockBridge`를 제공하므로, 단위 테스트에서
  명령어 응답을 스텁으로 대체할 수 있습니다.
</Callout>

## 아키텍처 한눈에 보기

웹 앱에서 온 요청은 브리지를 지나 네이티브 호스트로 내려가고, 코어로 들어간 다음, 번들 소스나 원격 서버로 나갑니다.

![웹 앱이 @wvb/bridge를 통해 네이티브 호스트를 호출하고, 호스트는 wvb Rust 코어를 임베드하며, 코어는 소스와 원격 서버에서 읽어 온다.](https://static.wvb.dev/diagrams/architecture.svg)

웹뷰도 같은 코어를 통해 자산을 로드합니다. 번들 프로토콜이 소스에서 바로 파일을 제공하므로, 페이지 바이트와 `invoke()` 명령어가 하나의 번들 소스를 공유합니다.
