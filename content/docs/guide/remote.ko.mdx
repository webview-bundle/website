---
title: 원격 서버 구축하기
description: 무선(OTA) 번들 전송을 위한 업데이트 서버를 세우고, 전체 흐름을 로컬에서 실행한 뒤 프로바이더를 선택하세요.
---

원격 서버는 작고 고정된 규약에 따라 앱의 업데이터에 번들을 제공하는 HTTP 서버입니다. [원격 번들](/docs/guide/remote-bundles)의 서버 측 짝에 해당합니다. 원격 서버를 만들고, 클라우드 계정 없이 로컬에서 배포와 업데이트 전체 흐름을 실행한 다음, 프로덕션용 프로바이더를 고르세요.

Webview Bundle은 `local`, `aws`, `cloudflare` 세 가지 백엔드용 원격 서버를 제공합니다. 각 백엔드는 역할에 따라 세 개의 패키지로 나뉩니다. 작업에 필요한 것만 사용하세요.

## 패키지 역할

모든 백엔드는 최대 세 개의 패키지를 가집니다.

| 역할            | 패키지                                  | 하는 일                                                                                                                        |
| --------------- | --------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| 설정 클라이언트 | `@wvb/remote-<backend>`                 | `{ uploader, deployer }`를 만드는 `wvb.config` 플러그인입니다. 퍼블리셔의 머신이나 CI에서 실행되어 번들을 푸시하고 배포합니다. |
| HTTP 서버       | `@wvb/remote-<backend>-provider`        | HTTP 규약을 구현하고 클라이언트에 번들을 제공하는 서버입니다.                                                                  |
| Pulumi IaC      | `@wvb/remote-<backend>-provider-pulumi` | 클라우드 리소스를 프로비저닝하고 프로바이더를 배포하는 코드형 인프라입니다.                                                    |

구체적으로, 백엔드별로 정리하면 다음과 같습니다.

| 백엔드       | 설정 클라이언트          | HTTP 서버                         | Pulumi IaC                               |
| ------------ | ------------------------ | --------------------------------- | ---------------------------------------- |
| `local`      | `@wvb/remote-local`      | `@wvb/remote-local-provider`      | —                                        |
| `aws`        | `@wvb/remote-aws`        | `@wvb/remote-aws-provider`        | `@wvb/remote-aws-provider-pulumi`        |
| `cloudflare` | `@wvb/remote-cloudflare` | `@wvb/remote-cloudflare-provider` | `@wvb/remote-cloudflare-provider-pulumi` |

설정 클라이언트는 `wvb.config.ts`의 `remote` 블록에 연결하는 대상입니다. `uploader`는 패킹된 `.wvb`를 스토리지에 쓰고, `deployer`는 클라이언트가 받아야 할 버전을 지정합니다. 그 결과를 `remote`에 펼쳐 넣으세요.

```ts title="wvb.config.ts"
import { defineConfig } from '@wvb/config';
import { awsRemote } from '@wvb/remote-aws';

export default defineConfig({
  remote: {
    endpoint: 'https://cdn.example.com',
    bundleName: 'app',
    ...awsRemote({ bucket: 'my-bundles', aws: { region: 'us-east-1' } }),
  },
});
```

퍼블리시 측 무결성과 서명 옵션은 `uploader`/`deployer`와 함께 위치합니다. [원격, 무결성 및 서명 설정](/docs/config/remote)을 참고하세요.

## HTTP 규약

모든 프로바이더는 동일한 네 개의 엔드포인트를 구현합니다. 따라서 클라이언트는 뒤에 어떤 백엔드가 있는지 신경 쓰지 않습니다.

| 메서드와 경로                   | 목적                           | 특이 사항                                                |
| ------------------------------- | ------------------------------ | -------------------------------------------------------- |
| `GET /bundles`                  | 배포된 번들 목록 조회          | `200` JSON `[{ "name", "version" }]`; 선택적 `?channel=` |
| `HEAD /bundles/{name}`          | 현재 버전의 메타데이터(헤더만) | 배포되지 않았으면 `404`                                  |
| `GET /bundles/{name}`           | 현재 버전 다운로드             | 배포되지 않았으면 `404`                                  |
| `GET /bundles/{name}/{version}` | 특정 버전 다운로드             | 다른 버전 허용이 아니면 `403`                            |

필수 응답 헤더는 `Webview-Bundle-Name`과 `Webview-Bundle-Version`입니다. 선택 헤더는 `Webview-Bundle-Integrity`, `Webview-Bundle-Signature`, `ETag`, `Last-Modified`입니다. 다운로드는 `Content-Type: application/webview-bundle`을 사용합니다. 상태 코드와 채널 쿼리 전체 명세는 [원격 번들](/docs/guide/remote-bundles)에 있습니다.

이 네 개의 라우트에 응답하는 최소한의 핸들러도 유효한 원격 서버입니다. 프로바이더 패키지는 필요하지 않습니다.

```ts title="minimal-remote.ts"
import { Hono } from 'hono';

const app = new Hono();

app.get('/bundles', c => c.json([{ name: 'app', version: '1.1.0' }]));

app.get('/bundles/:name', c => {
  const body = readCurrentBundle(c.req.param('name')); // 여러분의 스토리지
  return new Response(body, {
    headers: {
      'Content-Type': 'application/webview-bundle',
      'Webview-Bundle-Name': 'app',
      'Webview-Bundle-Version': '1.1.0',
    },
  });
});

app.get(
  '/bundles/:name/:version',
  c => c.body(null, 403) // 기본적으로 특정 버전은 금지
);

export default app;
```

## 로컬에서 흐름 테스트하기

업데이트 전체 흐름을 여러분의 머신에서 실행하세요. `@wvb/remote-local`은 업로드를 로컬 디렉터리(기본값 `~/.wvb/local`)에 쓰고, `wvb remote local`은 프로덕션 서버가 구현하는 것과 동일한 규약으로 그 디렉터리를 제공합니다.

### 설정을 로컬 프로바이더로 지정하기

```ts title="wvb.config.ts"
import { defineConfig } from '@wvb/config';
import { localRemote } from '@wvb/remote-local';

export default defineConfig({
  pack: {
    outFile: 'app', // → app.wvb
  },
  remote: {
    endpoint: 'http://localhost:4313',
    bundleName: 'app',
    ...localRemote(), // ~/.wvb/local에 쓰는 { uploader, deployer }
  },
});
```

홈 디렉터리 대신 프로젝트 안에 스토어를 두려면 다음과 같이 하세요.

```ts
...localRemote({ baseDir: './.wvb/local' }),
```

로컬 스토어는 번들과 버전별로 파일을 배치합니다.

```text
~/.wvb/local/bundles/app/app_1.1.0.wvb   # 번들 바이너리
~/.wvb/local/bundles/app/app_1.1.0.json  # { integrity?, signature? }
~/.wvb/local/bundles/app/deployment.json # 배포 포인터
```

### 버전 퍼블리시하기

```sh
wvb pack                                 # 설정으로부터 .wvb 빌드
wvb upload app --version 1.1.0 --deploy  # app_1.1.0.wvb를 ~/.wvb/local에 쓰고 배포
```

`wvb upload`는 기본적으로 업로드 전에 패킹하므로, 명시적인 `wvb pack`은 선택 사항입니다. `--deploy`의 기본값은 `false`입니다. 이 플래그를 전달하거나, 배포를 따로 실행하세요.

```sh
wvb deploy app --version 1.1.0           # 1.1.0을 클라이언트가 받는 버전으로 지정
```

버전은 위치 인자가 아니라 `--version,-V` 플래그입니다. 모든 플래그는 [CLI 레퍼런스](/docs/guide/cli)를 참고하세요.

### 제공하기

```sh
wvb remote local                         # ~/.wvb/local을 http://localhost:4313에서 제공
wvb remote local --base-dir ./.wvb/local --port 4313 --allow-other-versions
```

`wvb remote local`은 기본적으로 포트 `4313`(베이스 디렉터리 `~/.wvb/local`)에서 대기합니다. `--allow-other-versions`를 지정하면 클라이언트가 배포되지 않은 버전을 `403` 대신 `GET /bundles/{name}/{version}`로 가져올 수 있습니다.

### 앱을 원격 서버로 지정하기

앱의 업데이터 엔드포인트를 `http://localhost:4313`으로 설정하세요. 이제 `getUpdate`, `download`, `install`이 로컬 서버에 요청하며, 프로덕션과 똑같이 번들을 다운로드하고 검증하고 활성화합니다.

```rust
let remote = Remote::builder()
  .endpoint("http://localhost:4313")
  .build()?;
let updater = Updater::new(source, Arc::new(remote), None);

let info = updater.get_update("app").await?;   // HEAD /bundles/app
if info.is_available {
  updater.download("app", None).await?;       // GET /bundles/app
  updater.install("app", &info.version).await?; // 다운로드한 버전 활성화
}
```

<Callout type="info">
  Android 에뮬레이터에서는 호스트 머신이 `localhost`가 아닙니다. 업데이터를
  `http://10.0.2.2:4313`으로 지정하세요.
</Callout>

### 클라이언트 명령어로 확인하기

CLI는 어떤 원격 서버에 대해서도 클라이언트로 동작할 수 있습니다. 앱을 지정하기 전에 서버가 제대로 동작하는지 확인할 때 유용합니다.

```sh
wvb remote list     --endpoint http://localhost:4313  # GET  /bundles  (별칭: remote ls)
wvb remote current  app --endpoint http://localhost:4313  # HEAD /bundles/app
wvb remote download app --endpoint http://localhost:4313 --outfile ./app.wvb  # GET /bundles/app
```

### 단일 번들 미리 보기

서버나 배포 없이 `.wvb`를 살펴보려면, 그 파일들을 직접 제공하세요.

```sh
wvb serve ./app.wvb              # 번들의 파일들을 http://localhost:4312에서 제공
wvb serve ./app.wvb --port 8080
```

`wvb serve`는 기본적으로 포트 `4312`에서 대기합니다. 업로드하기 전에 패킹이 기대한 파일과 헤더를 만들었는지 확인할 때 사용하세요.

## 프로바이더 선택하기

로컬 테스트를 넘어서면 `localRemote()`를 클라우드 백엔드로 바꾸세요. 각 프로바이더는 `wvb.config.ts`용으로 호환되는 `uploader`와 `deployer`, 위 규약을 구현하는 서버, 그리고 이를 프로비저닝하는 Pulumi 패키지를 제공합니다.

```ts title="wvb.config.ts — Cloudflare"
import { cloudflareRemote } from '@wvb/remote-cloudflare';

export default defineConfig({
  remote: {
    endpoint: 'https://bundles.example.com',
    bundleName: 'app',
    ...cloudflareRemote({
      accountId: process.env.CF_ACCOUNT_ID!,
      bucket: 'webview-bundle',
      kvNamespaceId: process.env.CF_KV_NAMESPACE_ID!,
    }),
  },
});
```

Pulumi로 그에 맞는 인프라를 프로비저닝하세요.

```ts title="index.ts — pulumi up"
import { WebviewBundleRemoteProvider } from '@wvb/remote-cloudflare-provider-pulumi';

export const remote = new WebviewBundleRemoteProvider('app', {
  accountId: process.env.CF_ACCOUNT_ID!,
});
```
