title: Webview Bundle를 선택하는 이유
description: 웹 앱을 오프라인 우선 .wvb 번들로 출시하고 무선(OTA) 업데이트로 배포하는 방법을 설명합니다.
---

Webview Bundle는 빌드된 웹 앱을 하나의 압축된, 무결성이 검증되는 아카이브(`.wvb`)로 묶고, 이를 커스텀 URL 스킴을 통해 웹뷰에 제공합니다. 웹뷰는 HTML, JS, CSS, 미디어를 네트워크가 아닌 로컬 저장소에서 읽습니다. 원격을 설정하면 앱이 더 새로운 번들을 무선으로 다운로드하며, 네이티브 앱스토어 출시 없이도 갱신된 코드를 배포합니다.

팀이 Webview Bundle를 선택하는 이유는 세 가지입니다. 오프라인 우선 전달, 무선(OTA) 업데이트, 그리고 모든 웹뷰 플랫폼을 아우르는 하나의 워크플로입니다.

## 기본으로 오프라인 우선

원격 URL을 사용하면 첫 페인트가 네트워크를 기다려야 합니다. 캐시가 비어 있거나, 캡티브 포털에 걸리거나, 연결이 끊기면 빈 화면이 나타납니다.

Webview Bundle는 앱과 함께 출시되는 로컬 `.wvb`에서 자산을 곧바로 제공합니다. 프로토콜은 각 요청을 번들 안의 파일로 매핑하고 그 파일을 반환합니다.

```text
bundle://app/index.html  ->  index.html inside the app bundle
bundle://app/assets/x.js ->  assets/x.js inside the app bundle
```

```ts title="Serving a bundle locally"
import { BundleSource, BundleProtocol } from '@wvb/node';

const source = new BundleSource({
  builtinDir: '/path/to/builtin',
  remoteDir: '/path/to/remote',
});

const protocol = new BundleProtocol(source);

// 플랫폼 통합이 스킴 요청을 여기로 라우팅합니다.
const res = await protocol.handle('get', 'bundle://app/index.html');
// res: { status, headers, body }
```

첫 페인트는 절대 네트워크를 기다리지 않습니다. 지하철에서든, 비행기에서든, 세션 도중 연결이 끊겨도 앱은 로컬 번들에서 화면을 그립니다. 네트워크는 최적화 수단이 됩니다.

<Callout type="info">
  번들 프로토콜은 `GET`과 `HEAD`를 지원하고, 각 파일에 저장된 헤더를 그대로 재현하며, 미디어에 대한
  `Range` 요청에는 `206`으로 응답합니다. 전체 요청 매핑은 [프로토콜
  처리](/docs/guide/protocol-handling)를 참고하세요.
</Callout>

## 무선(OTA) 업데이트

앱스토어 출시는 느리고 심사에 막힙니다. 문구 오타 하나, 동작하지 않는 버튼 하나가 며칠에 걸친 심사에 묶일 수 있습니다. 앱 코드가 `.wvb` 안에 있기 때문에, 새 번들 버전을 배포하고 설치된 앱이 이를 무선(OTA)으로 다운로드하게 하면 문제를 고칠 수 있습니다. 네이티브 바이너리는 전혀 바뀌지 않습니다.

업데이트 모델은 두 개의 소스로 뒷받침됩니다. `builtin` 소스는 출시 시 함께 담은 읽기 전용 번들입니다. `remote` 소스는 다운로드한 번들을 담으며 우선권을 가집니다. 다운로드나 활성화가 실패하면 `builtin`으로 되돌아가므로, OTA가 기기를 동작하는 앱이 없는 상태로 만드는 일은 결코 없습니다.

```ts title="Two sources: remote wins, builtin is the floor"
new BundleSource({
  builtinDir: '/path/to/builtin', // 앱에 함께 담긴 읽기 전용
  remoteDir: '/path/to/remote', // OTA 다운로드, 우선권을 가짐
});
```

업데이트는 두 개의 명시적인 단계로 실행되어, 다운로드가 절반만 된 번들이 제공되는 일이 없습니다. `download`는 원격 소스에 스테이징하고 검증합니다. `install`은 스테이징된 버전을 활성화합니다.

```ts title="Check, download, install"
import { Updater } from '@wvb/node';

const updater = new Updater(source, remote);

const info = await updater.getUpdate('app');
if (info.isAvailable) {
  await updater.download('app'); // 스테이징 + 검증
  await updater.install('app', info.version); // 활성화
}
```

### 채널을 이용한 단계적 롤아웃

채널은 목록 및 다운로드 요청에 `?channel=`로 전달되는 선택적 문자열입니다. `UpdaterOptions`를 통해 전달하면 특정 기기 집단을 `beta`로 지정한 뒤 승격할 수 있습니다. 생략하면 이 파라미터는 빠지며, 하드코딩된 채널 이름은 없습니다.

```ts title="Pin this device to the beta channel"
const updater = new Updater(source, remote, { channel: 'beta' });
```

```http
GET /bundles/app?channel=beta
```

### 무결성과 서명을 통한 진위 확인

모든 다운로드는 활성화 전에 검증됩니다. 무결성은 SHA-2(`sha256` 기본값, 그 외 `sha384`, `sha512`)를 사용하며 `"<alg>:<base64>"` 형식으로 직렬화됩니다.

```text
sha256:n4bQgYhMfWWaL+qgxVrQFaO/TxsrC4Is0V1sFbDwCgg=
```

기본 정책은 `optional`(값이 있으면 검증)입니다. `strict`로 설정하면 무결성을 필수로 요구합니다. 여기에 선언적 `signatureVerifier`를 추가하면 누가 게시했는지 증명할 수 있습니다. 서명은 무결성 문자열의 바이트를 대상으로 합니다.

```ts title="Require integrity, verify an Ed25519 signature"
const updater = new Updater(source, remote, {
  integrityPolicy: 'strict',
  signatureVerifier: {
    algorithm: 'ed25519',
    key: { format: 'raw', data: publicKeyBytes }, // 32바이트 Ed25519 키
  },
});
```

검증은 ECDSA(`ecdsaSecp256R1`, `ecdsaSecp384R1`), `ed25519`, RSA(`rsaPkcs1V15`, `rsaPss`)를 지원합니다.

<Callout type="info">
  무결성은 SHA-3가 아니라 SHA-2입니다. 서명은 원시 아카이브 바이트가 아니라 무결성 문자열을 대상으로
  하며, 무결성 값이 반드시 존재해야 합니다. 검증 흐름은 [원격 번들](/docs/guide/remote-bundles)을,
  스키마는 [원격, 무결성 및 서명 설정](/docs/config/remote)을 참고하세요.
</Callout>

## 웹 개발자를 위해

이미 알고 있는 방식 그대로 앱을 작성하세요. Webview Bundle는 소스가 아니라 번들러의 출력물을 대상으로 동작하므로 React, Vue, Svelte, 순수 HTML 모두 그대로 동작합니다. 커스텀 프레임워크도, 특별한 빌드 단계도 필요 없습니다. `wvb pack`을 빌드 디렉터리로 향하게 하세요.

```sh title="Pack your build output into one .wvb"
npm install --save-dev @wvb/cli
npx wvb pack ./dist
```

패킹된 파일은 기본적으로 `.wvb/<pkgName>`에 생성되며 `.wvb` 확장자는 자동으로 붙습니다. `outFile` 필드(단일 경로)로 이를 재정의할 수 있습니다.

```ts title="wvb.config.ts"
import { defineConfig } from '@wvb/config';

export default defineConfig({
  pack: {
    outFile: '.wvb/app',
  },
});
```

웹뷰 안에서 코드는 `@wvb/bridge`를 통해 네이티브 호스트와 통신합니다. 하나의 `invoke()` API가 소스, 원격, 업데이터 명령을 노출하고, 그 아래에서 Electron, Tauri, Android, iOS에 맞는 전송 방식을 선택합니다. 같은 업데이트 코드가 어디서나 실행됩니다.

```ts title="Driving updates from web code"
import { updater } from '@wvb/bridge';

const update = await updater.getUpdate('app');
if (update.isAvailable) {
  await updater.download('app');
  await updater.install('app', update.version);
}
```

브리지는 호출 시점에 플랫폼을 확인하므로, 직접 플랫폼별로 분기할 필요가 없습니다.

```ts title="Platform is detected for you"
import { platform } from '@wvb/bridge';

platform.type; // 'electron' | 'tauri' | 'android' | 'ios'
```

### 다른 방식과의 비교

흔히 쓰이는 두 가지 대안이 같은 문제의 일부를 해결합니다.

- **플랫폼별 네이티브 자산 번들링.** 웹 빌드를 각 셸에 임베드하고, 서빙 및 업데이트 로직을 플랫폼마다 한 번씩 작성하는 방식입니다. 동작은 하지만 툴체인, 자산 포맷, 업데이트 코드가 플랫폼마다 다르고 그 모두를 직접 유지보수해야 합니다. Webview Bundle는 공유된 Rust 코어를 기반으로 Electron, Tauri, Android, iOS 전반에 하나의 `.wvb`와 하나의 사고 모델을 제공합니다.
- **CodePush 방식의 OTA.** 호스팅형 OTA 서비스는 스토어 심사 없이 JS 업데이트를 배포합니다. 원격 소스와 같은 이점을 줍니다. 다만 절충점이 다릅니다. Webview Bundle는 관리형 서드파티 서비스가 아니라, builtin 폴백, 채널 롤아웃, 무결성과 선택적 서명을 갖춘, 직접 호스팅하는 검증 가능한 번들 포맷입니다. 서버와 아티팩트를 여러분이 소유합니다.

솔직한 절충점은 이렇습니다. 번들 포맷과 패킹 단계를 도입하고 원격을 직접 운영해야 합니다. 그 대가로 오프라인 우선 전달, 스토어 심사 없는 OTA, 그리고 플랫폼을 추가해도 변하지 않는 하나의 워크플로를 얻습니다.

<Callout type="info">
  Android와 iOS는 구현되어 출시된 상태입니다. iOS 최소 버전은 iOS 16입니다. Deno Desktop은
  실험적입니다. 현재 상태는 [플랫폼 지원](/docs/guide/platform-support)을 참고하세요.
</Callout>
