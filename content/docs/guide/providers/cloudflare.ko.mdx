---
title: Cloudflare 프로바이더
description: R2, Workers KV, Cloudflare Workers를 사용해 Cloudflare에서 번들을 저장하고 배포하고 제공합니다.
---

Webview Bundle 원격을 Cloudflare 엣지에서 완전히 실행할 수 있습니다. 번들은 **R2**에, 버전 포인터는 **Workers KV**에 저장하고, **Cloudflare Worker**가 업데이터와 통신하는 HTTP 계약을 제공합니다. 관리할 서버 없이 전 세계에 낮은 지연으로 무선(OTA) 전달을 제공합니다.

세 개의 패키지가 이 워크플로를 담당합니다.

| Package                                  | 역할                                    | 실행 위치          |
| ---------------------------------------- | --------------------------------------- | ------------------ |
| `@wvb/remote-cloudflare`                 | `wvb.config`용 업로더 + 디플로이어      | 로컬 머신 / CI     |
| `@wvb/remote-cloudflare-provider`        | 번들을 제공하는 Worker                  | Cloudflare Workers |
| `@wvb/remote-cloudflare-provider-pulumi` | 인프라를 프로비저닝하는 Pulumi 컴포넌트 | `pulumi up`        |

세 패키지 모두 `0.1.0`입니다. `wvb.config`에서 버전을 고정하세요.

```sh
npm i -D @wvb/remote-cloudflare
```

<Callout type="info">
  원격이 처음이신가요? [원격 만들기](/docs/guide/remote)에서 계약과 로컬 테스트 루프를 먼저
  살펴보세요. 전체 `wvb.config` 레퍼런스는 [원격, 무결성 및 서명 설정](/docs/config/remote)을
  참고하세요.
</Callout>

## 각 구성 요소가 맞물리는 방식

각 역할은 하나의 Cloudflare 서비스에 매핑됩니다.

- **R2**는 R2의 S3 호환 API를 통해 각 번들을 객체로 저장합니다. `@wvb/remote-cloudflare` 업로더는 `@wvb/remote-aws`의 S3 업로더를 R2로 향하게 하여 재사용합니다.
- **Workers KV**는 배포 포인터를 보관합니다. 번들마다(또는 번들과 채널마다) 키 하나를 두고, 그 값은 배포된 버전입니다.
- `@wvb/remote-cloudflare-provider`를 실행하는 **Cloudflare Worker**는 KV를 읽어 버전을 확인한 뒤, R2에서 일치하는 객체를 스트리밍합니다.

<Callout type="warn">
  Cloudflare에는 기본 서명 기능이 없습니다. KMS에 해당하는 것이 없고, `@wvb/remote-cloudflare`도
  서명자를 제공하지 않습니다. [서명](/docs/config/remote)이 필요하면 다른 곳에서 생성한 뒤, 결과로
  나온 `signature` 문자열을 `wvb.config`를 통해 전달하세요.
</Callout>

## 업로더와 디플로이어 설정

`cloudflareRemote()`를 호출하고 그 결과를 `remote` 블록에 펼쳐 넣으세요. 이 함수는 `{ uploader, deployer }`를 반환합니다. 업로더는 번들을 R2에 쓰고, 디플로이어는 버전을 KV에 씁니다.

```ts title="wvb.config.ts"
import { defineConfig } from '@wvb/config';
import { cloudflareRemote } from '@wvb/remote-cloudflare';

export default defineConfig({
  remote: {
    endpoint: 'https://bundles.example.com',
    bundleName: 'my-app',
    ...cloudflareRemote({
      bucket: 'webview-bundle',
      accountId: process.env.CLOUDFLARE_ACCOUNT_ID!,
      kvNamespaceId: process.env.CLOUDFLARE_KV_NAMESPACE_ID!,
    }),
  },
});
```

| Option          | Type     | 설명                                                                                  |
| --------------- | -------- | ------------------------------------------------------------------------------------- |
| `bucket`        | `string` | 번들 객체를 저장하는 R2 버킷.                                                         |
| `accountId`     | `string` | Cloudflare 계정 ID. R2 엔드포인트와 KV 대상을 설정합니다.                             |
| `kvNamespaceId` | `string` | 배포 포인터를 보관하는 Workers KV 네임스페이스.                                       |
| `uploader`      | object   | 내부 S3 업로더에 대한 선택적 재정의(`key`, `contentType`, `metadata` 등).             |
| `deployer`      | object   | KV 디플로이어에 대한 선택적 재정의(`key`, `expiration`, `expirationTtl`, `metadata`). |

기본값: 업로더는 `region: 'auto'`와 엔드포인트 `https://<accountId>.r2.cloudflarestorage.com`을 설정한 다음, `bundles/<name>/<name>_<version>.wvb` 키에 객체를 저장하고 `webview-bundle-*` 커스텀 메타데이터를 붙입니다. 디플로이어는 `<name>` 키(채널 배포일 때는 `<name>/<channel>`)에 버전을 KV로 씁니다.

레이아웃이 다르면 둘 중 어느 쪽이든 재정의하세요.

```ts title="wvb.config.ts"
...cloudflareRemote({
  bucket: 'webview-bundle',
  accountId: process.env.CLOUDFLARE_ACCOUNT_ID!,
  kvNamespaceId: process.env.CLOUDFLARE_KV_NAMESPACE_ID!,
  uploader: {
    key: (name, version) => `releases/${name}/${version}.wvb`,
    cacheControl: 'public, max-age=31536000, immutable',
  },
  deployer: {
    key: (name, _version, channel) => (channel ? `${name}/${channel}` : name),
    expirationTtl: 60 * 60 * 24,
  },
}),
```

R2에는 S3 자격 증명이 필요합니다. AWS와 동일한 방식으로 제공하세요. S3 클라이언트가 읽는 환경변수를 통해 전달합니다.

```sh
export AWS_ACCESS_KEY_ID=<r2-access-key-id>
export AWS_SECRET_ACCESS_KEY=<r2-secret-access-key>
```

설정을 마쳤으면 번들을 게시하세요.

```sh
wvb pack ./dist
wvb upload --version 1.2.0
wvb deploy --version 1.2.0
```

`wvb upload`는 기본적으로 배포하지 않습니다. `wvb deploy`를 실행하거나(또는 `upload`에 `--deploy`를 전달하여) 해당 버전을 라이브 상태로 만드세요.

```sh
wvb upload --version 1.2.0 --deploy
```

기본 포인터 대신 채널로 배포하려면 다음과 같이 합니다.

```sh
wvb deploy --version 1.2.0 --channel beta
```

모든 플래그는 [CLI 레퍼런스](/docs/guide/cli)를 참고하세요.

## Worker로 번들 제공하기

제공 측은 `@wvb/remote-cloudflare-provider`의 Hono 앱입니다. `wvbRemote()`로 생성한 뒤 Worker의 바인딩을 그 `Context`로 전달하세요. `KV` 바인딩은 `kv`가 되고, `BUCKET`(R2) 바인딩은 `r2`가 됩니다.

```ts title="src/worker.ts"
import { wvbRemote } from '@wvb/remote-cloudflare-provider';

const remote = wvbRemote();

export interface Env {
  KV: KVNamespace;
  BUCKET: R2Bucket;
}

export default {
  fetch(request: Request, env: Env): Response | Promise<Response> {
    return remote.fetch(request, { kv: env.KV, r2: env.BUCKET });
  },
};
```

`wvbRemote()`는 옵션 하나 `allowOtherVersions`(기본값 `false`)를 받습니다. `false`일 때 `GET /bundles/:name/:version`은 `403`을 반환합니다. 배포된 버전 외의 버전도 클라이언트가 다운로드하게 하려면 `true`로 설정하세요.

```ts title="src/worker.ts"
const remote = wvbRemote({ allowOtherVersions: true });
```

Worker는 KV에서 버전을 확인하고, R2에서 `bundles/<name>/<name>_<version>.wvb`를 읽어, 필수 헤더인 `Webview-Bundle-Name`과 `Webview-Bundle-Version`(객체 메타데이터에 있으면 `Webview-Bundle-Integrity`와 `Webview-Bundle-Signature`도 함께)을 붙여 반환합니다.

이름이 `Context`와 맞도록 `wrangler.jsonc`에서 `KV`와 `BUCKET`을 바인딩하세요.

```json title="wrangler.jsonc"
{
  "name": "webview-bundle",
  "main": "src/worker.ts",
  "compatibility_date": "2025-01-01",
  "kv_namespaces": [{ "binding": "KV", "id": "<kv-namespace-id>" }],
  "r2_buckets": [{ "binding": "BUCKET", "bucket_name": "webview-bundle" }]
}
```

Worker를 배포하세요.

```sh
npx wrangler deploy
```

라이브 엔드포인트에 대해 계약을 검증하세요.

```sh
curl https://bundles.example.com/bundles
# [{ "name": "my-app", "version": "1.2.0" }]

curl -I https://bundles.example.com/bundles/my-app
# Webview-Bundle-Name: my-app
# Webview-Bundle-Version: 1.2.0
# Content-Type: application/webview-bundle
```

## Pulumi로 프로비저닝하기

`@wvb/remote-cloudflare-provider-pulumi`는 전체 스택을 하나의 컴포넌트 리소스로 프로비저닝합니다. R2 버킷, Workers KV 네임스페이스, Worker(버전과 배포 포함)를 만들고, `Context`가 기대하는 바인딩 `BUCKET`(R2)과 `KV`로 서로 연결합니다.

```ts title="index.ts"
import { WebviewBundleRemoteProvider } from '@wvb/remote-cloudflare-provider-pulumi';

const provider = new WebviewBundleRemoteProvider('webview-bundle', {
  accountId: process.env.CLOUDFLARE_ACCOUNT_ID!,
  // Worker를 점진적으로 롤아웃합니다. 기본값은 100입니다.
  workerDeploymentPercentage: 25,
});

export const bucketName = provider.bucketName;
export const kvNamespaceId = provider.kvNamespaceId;
export const workerId = provider.workerId;
export const workerVersionId = provider.workerVersionId;
export const workerDeploymentId = provider.workerDeploymentId;
```

이 컴포넌트는 Pulumi 리소스 토큰 `webview-bundle:cloudflare:RemoteProvider`를 등록합니다. `workerDeploymentPercentage`는 새 Worker 버전이 받는 트래픽 비율입니다(기본값 `100`).

```sh
pulumi up
pulumi stack output kvNamespaceId
```

내보낸 `bucketName`과 `kvNamespaceId`를 위의 `cloudflareRemote()`에 넣은 다음, CLI로 게시하세요.
