---
title: 웹뷰 번들 (.wvb)
---

웹뷰 번들은 여러 웹 리소스를 `.wvb` 파일 하나로 묶는 아카이브 포맷입니다. 웹 애플리케이션을 패킹하면 HTML·JS·CSS·이미지 같은 자산이 이 파일 하나에 담깁니다. 네이티브 호스트는 웹뷰의 요청을 가로채 이 파일에서 자산을 읽어 응답합니다. 각 자산은 체크섬과 함께 저장되므로, 네트워크 없이도 무결성이 검증된 자산을 제공할 수 있습니다.

## 데이터 구조

![.wvb 파일은 헤더, 인덱스, 데이터 세 부분으로 이루어지며, 각 부분은 xxHash-32 체크섬으로 끝납니다.](/diagrams/wvb-structure.svg)

`.wvb` 파일은 헤더, 인덱스, 데이터 세 부분으로 구성됩니다. 세 부분은 각각 xxHash-32 체크섬으로 끝나며, 이 값으로 무결성을 검증합니다.

### 헤더

헤더는 파일의 첫 17바이트로, 포맷 메타데이터를 담습니다.

- **매직 넘버 (8바이트)** — 웹뷰 번들 포맷을 식별하는 고정값입니다. `0xF09F8C90F09F8E81`을 사용합니다. 이는 UTF-8로 "🌐🎁"이며 빅엔디언으로 기록됩니다.
- **버전 (1바이트)** — 포맷 버전입니다. 현재 값은 v1(`0x01`)입니다.
- **인덱스 크기 (4바이트)** — 인덱스 부분의 바이트 크기입니다. u32 값을 빅엔디언으로 저장합니다.
- **체크섬 (4바이트)** — 앞의 13바이트가 온전한지 검증하는 xxHash-32 해시입니다.

### 인덱스

인덱스는 요청 경로를 데이터 위치에 1:1로 대응시키는 맵입니다. 호스트는 요청 경로로 인덱스를 조회해 해당 자산의 데이터 범위를 찾고, 그 범위를 읽어 응답합니다.

![요청 경로를 인덱스에서 조회해 오프셋과 길이를 얻고, 그 범위의 바이트를 데이터에서 읽습니다.](/diagrams/wvb-index-lookup.svg)

- 인덱스는 헤더에 명시된 크기만큼 차지하며, 빅엔디언으로 인코딩됩니다.
- 경로(key)를 항목(value)에 대응시키는 맵이며, 결과가 항상 같도록 key를 정렬한 순서로 직렬화됩니다.
- 각 항목은 다음 값을 담습니다.
  - **오프셋, 길이** — 데이터에서 파일 바이트를 읽을 범위를 지정합니다.
  - **콘텐츠 타입, 콘텐츠 길이** — 자산의 MIME 타입과 압축 전 원본 크기입니다.
  - **HTTP 헤더** — 프로토콜로 응답할 때 함께 내보낼 헤더입니다. (선택)
- 마지막 4바이트는 인덱스가 온전한지 검증하는 xxHash-32 체크섬입니다.

### 데이터

데이터는 각 항목의 파일 바이트를 순서대로 저장합니다. 호스트는 인덱스에서 얻은 오프셋과 길이로 읽을 범위를 지정해 해당 자산만 읽습니다.

![데이터는 각 항목의 압축된 바이트를 이어 붙여 저장하고, 인덱스가 각 항목의 오프셋과 길이를 기록합니다.](/diagrams/wvb-data-layout.svg)

예를 들어 `/index.js` 항목이 오프셋 `0`, 길이 `768`로 기록되어 있으면, 호스트는 데이터의 `0`부터 `768`바이트까지를 읽습니다.

- 각 항목은 [LZ4 블록 포맷](https://github.com/lz4/lz4/blob/dev/doc/lz4_Block_format.md)으로 압축됩니다.
- 마지막 4바이트는 데이터가 온전한지 검증하는 xxHash-32 체크섬입니다.

## 설계 결정

### `.zip` 대신 새 포맷을 만든 이유

`.zip`은 파일 목록(중앙 디렉터리)을 파일 끝에 두므로, 자산을 찾으려면 먼저 끝을 읽어야 합니다. 또한 자산을 HTTP로 응답하는 데 필요한 콘텐츠 타입이나 헤더 같은 정보를 담지 않습니다.

웹뷰 번들은 이 포맷을 웹뷰에 자산을 제공하는 용도에 맞춰 설계했습니다.

- **인덱스를 앞에 둡니다.** 파일을 스캔하지 않고, 한 번의 조회로 경로에서 데이터 위치를 얻습니다.
- **항목마다 HTTP 메타데이터를 저장합니다.** 호스트는 콘텐츠 타입과 헤더를 그대로 HTTP 응답으로 내보냅니다.
- **바이트 배치를 고정합니다.** 오프셋과 길이로 필요한 범위만 읽으므로, 부분 요청(HTTP range)과 스트리밍에 유리합니다.
- **가벼운 체크섬으로 검증합니다.** 헤더·인덱스·데이터를 각각 xxHash-32로 검증합니다.

### LZ4를 선택한 이유

자산 압축 해제는 웹뷰에 응답하는 경로에서 매번 일어나므로, 압축률보다 해제 속도가 중요합니다.

- **압축 해제가 매우 빠릅니다.** 첫 화면을 그리는 순간을 지연시키지 않습니다.
- **CPU를 적게 씁니다.** UI 스레드를 막지 않습니다.
- **항목마다 독립적으로 압축합니다.** [LZ4 블록 포맷](https://github.com/lz4/lz4/blob/dev/doc/lz4_Block_format.md)으로 각 자산을 따로 해제하므로, 필요한 자산만 임의 접근으로 읽을 수 있습니다.
