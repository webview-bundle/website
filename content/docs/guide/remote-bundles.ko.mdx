Webview Bundle은 새 번들을 무선(OTA)으로 다운로드합니다. 따라서 네이티브 앱스토어 배포 없이도 업데이트된 앱 코드를 배포할 수 있습니다. 이 페이지에서는 업데이트 라이프사이클, 언어에 종속되지 않는 업데이터 모델, 모든 원격 서버가 구현하는 HTTP 규약, 그리고 무결성과 서명이 각 번들의 신뢰성을 지키는 방법을 다룹니다. 서버를 구축하려면 [원격 서버 구축하기](/docs/guide/remote)를 참고하세요. 언어별 업데이터 API는 [Node](/docs/references/node)와 [Deno](/docs/references/deno) 레퍼런스를 참고하세요.

## 업데이트 라이프사이클

번들은 빌드 산출물에서 기기까지 다섯 단계를 거칩니다. 앞의 세 단계는 사용자의 머신이나 CI에서 실행되고, 마지막 두 단계는 업데이터가 앱 내부에서 구동합니다.

![개발자 머신이 번들을 원격 서버로 업로드하고 배포하면, 기기가 업데이트를 확인하고 다운로드하고 검증한 뒤 설치합니다.](https://static.wvb.dev/diagrams/update-lifecycle.svg)

1. 빌드 산출물을 `.wvb` 아카이브로 **패킹**합니다.
2. 해당 아카이브를 **업로드**합니다. 무결성 해시와 서명을 함께 첨부할 수도 있습니다.
3. 업로드한 버전을 **배포**하면 해당 채널의 클라이언트가 이를 보기 시작합니다.
4. 기기에서 업데이터가 새 버전을 **확인**한 뒤 **다운로드**합니다.
5. 업데이터가 다운로드를 **검증**하고 이를 `remote` 소스에 **설치**합니다.

1~3단계는 CLI 명령어입니다.

```sh
wvb pack                                   # 빌드 산출물 -> .wvb/<pkgName>
wvb upload --version 1.2.0                 # 서버로 푸시 (배포하지 않음)
wvb deploy --version 1.2.0                 # 1.2.0을 현재 버전으로 지정
```

`wvb upload --deploy`(기본값 `false`)는 업로드와 배포를 한 번에 처리합니다. 모든 플래그는 [CLI 레퍼런스](/docs/guide/cli)를 참고하세요. 4~5단계는 업데이터이며, 다음에서 다룹니다.

## 업데이터 모델

업데이터는 [번들 소스](/docs/guide/bundle-sources)를 원격 엔드포인트에 연결하고 세 가지 작업을 제공합니다. 이 작업들은 분리되어 있어, 기기가 언제 바이트를 가져올지와 언제 버전을 전환할지를 정확히 제어할 수 있습니다.

- **`getUpdate`** — 확인만 합니다. 서버에 현재 배포된 버전을 요청하고 이를 로컬 버전과 비교합니다. 로컬 버전이 없거나 로컬 버전이 배포된 버전과 다르면 `isAvailable`을 `true`로 보고합니다. 아무것도 다운로드하지 않습니다.
- **`download`** — 번들을 가져와 무결성 및 서명 정책에 따라 검증하고 `remote` 소스에 스테이징합니다. 새 버전을 활성화하지는 **않습니다**.
- **`install`** — 스테이징된 번들을 디스크에서 다시 검증하고, 현재 버전으로 활성화하고, 이전 디스크립터를 언로드하고, 더 이상 유지하지 않는 버전을 정리합니다.

이렇게 분리하면 백그라운드에서 다운로드하고 다음 앱 실행 시 설치할 수 있습니다.

<Callout type="info">
  `builtin` 소스 — 앱 내부에 포함되어 배포되는 번들 — 는 항상 폴백으로 남아 있습니다. 버전이
  설치되면 `remote` 소스가 우선하므로, 다운로드된 업데이트가 builtin 번들보다 우선합니다. 원격
  버전이 없으면 앱은 builtin으로 폴백합니다. [번들 소스](/docs/guide/bundle-sources)를 참고하세요.
</Callout>

Rust 코어 흐름입니다. 플랫폼 래퍼는 이를 일대일로 그대로 따릅니다.

```rust
use std::sync::Arc;
use wvb::updater::Updater;

let updater = Updater::new(source, remote, None); // None -> UpdaterConfig::default()

let update = updater.get_update("app").await?;
if update.is_available {
  // download는 가져오고, 검증하고, 스테이징하지만 활성화하지는 않음
  let info = updater.download("app", None).await?;
  // install은 다시 검증하고, 활성화하고, 이전 버전을 정리함
  updater.install("app", info.version).await?;
  // 새 버전을 제공하도록 웹뷰를 리로드
}
```

`get_update`는 `BundleUpdateInfo`를 반환합니다.

```rust
pub struct BundleUpdateInfo {
  pub name: String,
  pub version: String,
  pub local_version: Option<String>,
  pub is_available: bool,
  pub etag: Option<String>,
  pub integrity: Option<String>,
  pub signature: Option<String>,
  pub last_modified: Option<String>,
}
```

`download`에 `Some(version)`을 전달하면 현재 버전 대신 특정 버전을 가져옵니다.

```rust
// 현재 배포된 버전을 가져옴
let info = updater.download("app", None).await?;
// 정확한 버전을 가져옴 (GET /bundles/{name}/{version})
let info = updater.download("app", Some("1.1.0".into())).await?;
```

플랫폼 가이드에는 각 대상에 맞는 관용적인 호출 방법이 나와 있습니다.
[Electron](/docs/guide/platforms/electron), [Tauri](/docs/guide/platforms/tauri),
[Android](/docs/guide/platforms/android), [iOS](/docs/guide/platforms/ios),
[Deno Desktop](/docs/guide/platforms/deno)을 참고하세요.

## 원격 HTTP 규약

Webview Bundle 서버는 네 개의 엔드포인트를 구현합니다. 프로바이더 패키지(local, AWS, Cloudflare)가 이 명세를 구현하며, 이에 맞춰 직접 구현할 수도 있습니다.

| 메서드와 경로                   | 목적                                        |
| ------------------------------- | ------------------------------------------- |
| `GET /bundles`                  | 배포된 번들 목록: `[{ "name", "version" }]` |
| `HEAD /bundles/{name}`          | 현재 버전의 메타데이터 (헤더만, 본문 없음)  |
| `GET /bundles/{name}`           | 현재 버전 다운로드                          |
| `GET /bundles/{name}/{version}` | 특정 버전 다운로드                          |

`GET /bundles`, `HEAD /bundles/{name}`, `GET /bundles/{name}`은 선택적인 `?channel=` 쿼리를 받습니다. `GET /bundles/{name}/{version}`은 받지 않습니다. 메타데이터는 응답 헤더로 전달됩니다.

| 헤더                       | 필수 여부 | 의미          |
| -------------------------- | --------- | ------------- |
| `Webview-Bundle-Name`      | 예        | 번들 이름     |
| `Webview-Bundle-Version`   | 예        | 버전          |
| `Webview-Bundle-Integrity` | 아니오    | 무결성 문자열 |
| `Webview-Bundle-Signature` | 아니오    | 서명          |
| `ETag`, `Last-Modified`    | 아니오    | 표준 검증자   |

다운로드 응답은 다음과 같습니다.

```sh
$ curl -sI http://localhost:4313/bundles/app
HTTP/1.1 200 OK
Content-Type: application/webview-bundle
Webview-Bundle-Name: app
Webview-Bundle-Version: 1.2.0
Webview-Bundle-Integrity: sha256:n4bQgYhMfWWaL+qgxVrQFaO/TxsrC4Is0V1sFbDwCgg=
ETag: "abc123"
```

목록 엔드포인트는 `application/json`을 반환합니다.

```json
[
  { "name": "app", "version": "1.2.0" },
  { "name": "admin", "version": "0.4.1" }
]
```

번들이나 버전이 없으면 `404`를 반환합니다. 서버가 허용하지 않는 경우, 배포되지 않은 특정 버전을 요청하면 `403`을 반환합니다. 필수 헤더가 없거나 2xx가 아닌 응답이 오면 클라이언트는 다운로드를 거부합니다. 기본 요청 타임아웃은 120초입니다. 실제 서버 예제와 로컬 테스트는 [원격 서버 구축하기](/docs/guide/remote)를 참고하세요.

## 무결성

무결성 해시를 사용하면 기기가 다운로드한 바이트가 게시한 바이트와 정확히 일치하는지 확인할 수 있습니다. Webview Bundle은 **SHA-2**를 사용합니다. `sha256`(기본값), `sha384`, `sha512`를 지원하며, `"<alg>:<base64>"` 형식으로 직렬화됩니다.

```text
sha256:n4bQgYhMfWWaL+qgxVrQFaO/TxsrC4Is0V1sFbDwCgg=
```

게시자는 업로드 시점에 이 문자열을 첨부하고([원격 설정](/docs/config/remote)), 서버는 이를 `Webview-Bundle-Integrity`로 반환하며, 업데이터는 다운로드 시 **무결성 정책**에 따라 이를 적용합니다.

| 정책       | 동작                                                          |
| ---------- | ------------------------------------------------------------- |
| `Strict`   | 무결성 값이 반드시 존재하고 일치해야 하며, 그렇지 않으면 거부 |
| `Optional` | **기본값** — 존재하면 검증하고, 첨부되지 않았으면 허용        |
| `None`     | 무결성 검사를 완전히 건너뜀                                   |

```rust
use wvb::integrity::IntegrityPolicy;
use wvb::updater::UpdaterConfig;

let config = UpdaterConfig::new()
  .integrity_policy(IntegrityPolicy::Strict);
```

## 서명

무결성이 바이트가 손상되지 않았음을 증명한다면, 서명은 _누가_ 게시했는지를 증명합니다. 서명은 원본 번들이 아니라 **무결성 문자열 바이트**(`"<alg>:<base64>"`)를 대상으로 합니다. 따라서 서명 검증에는 무결성 값이 반드시 존재해야 합니다. 공개 키를 제공하면, 업데이터는 서명이 검증되지 않는 모든 다운로드를 거부합니다.

검증 알고리즘은 다음과 같습니다.

- **ECDSA** — 곡선 P-256(secp256r1) 또는 P-384(secp384r1)
- **Ed25519**
- **RSA** — PKCS#1 v1.5 또는 PSS 패딩, SHA-256으로 해싱

공개 키 형식은 다음과 같습니다.

| 형식             | 로더                      | 적용 대상    |
| ---------------- | ------------------------- | ------------ |
| SPKI PEM         | `from_public_key_pem`     | 전체         |
| SPKI DER         | `from_public_key_der`     | 전체         |
| PKCS#1 PEM/DER   | `from_pkcs1_pem` / `_der` | RSA 전용     |
| SEC1 point bytes | `from_sec1_bytes`         | ECDSA 전용   |
| Raw 32-byte key  | `from_public_key_bytes`   | Ed25519 전용 |

일치하는 무결성 해시와 유효한 Ed25519 서명을 모두 요구하는 업데이터입니다.

```rust
use wvb::integrity::IntegrityPolicy;
use wvb::signature::{Ed25519Verifier, SignatureVerifier};
use wvb::updater::{Updater, UpdaterConfig};
use std::sync::Arc;

let verifier = SignatureVerifier::Ed25519(Arc::new(
  Ed25519Verifier::from_public_key_pem(PUBLIC_KEY_PEM)?,
));
let config = UpdaterConfig::new()
  .integrity_policy(IntegrityPolicy::Strict)
  .signature_verifier(verifier);
let updater = Updater::new(source, remote, Some(config));
```

SPKI PEM에서 로드한 ECDSA P-256입니다.

```rust
use wvb::signature::{EcdsaSecp256r1Verifier, SignatureVerifier};
use std::sync::Arc;

let verifier = SignatureVerifier::EcdsaSecp256r1(Arc::new(
  EcdsaSecp256r1Verifier::from_public_key_pem(PUBLIC_KEY_PEM)?,
));
```

게시 측 대응 항목 — 어떤 알고리즘으로 서명할지와 개인 키 형식 — 은 [원격 설정](/docs/config/remote)에서 지정합니다.

## 채널

채널은 단계적 롤아웃을 위해 서로 다른 버전을 서로 다른 대상에게 라우팅합니다. 채널에 배포하면 해당 채널을 요청하는 클라이언트만 이를 수신합니다. 채널이 없는 클라이언트는 기본 배포를 받습니다.

```sh
wvb upload --version 1.2.0 --deploy --channel beta   # beta 클라이언트만 1.2.0을 봄
wvb deploy --version 1.2.0                            # 나중에 기본 채널로 승격
```

기기에서 업데이터는 설정에 채널을 담아 `?channel=`로 전달합니다.

```rust
let config = UpdaterConfig::new().channel("beta");
let updater = Updater::new(source, remote, Some(config));
// 이제 get_update / download는 ?channel=beta를 요청함
```

채널은 자유 형식 문자열입니다. 하드코딩된 기본 이름은 없으며, 채널이 설정되지 않으면 쿼리 파라미터가 생략됩니다.
